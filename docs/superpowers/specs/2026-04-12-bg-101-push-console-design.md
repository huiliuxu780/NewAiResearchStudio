# BG-101 推送模块前端管理台设计

## 背景

后端 `push` 域已经具备较完整的 API 能力，覆盖：

- `channels`
- `tasks`
- `records`
- `templates`
- `stats`
- `events/trigger`

但前端目前没有任何对应入口、页面或组件。也就是说，推送模块的当前状态是“后端已具备管理能力，前端仍为空白”。

这次目标不是一口气做成完整运营后台，而是先把它收口成一个 **最小可用的推送管理台**。

用户已确认本轮采用：

- **方案 B：单页标签工作台**
- 并且 **需要挂进配置中心**

---

## 目标

1. 新增一个可用的推送管理页，接入真实后端 API
2. 页面结构围绕 `渠道 / 任务 / 记录 / 模板` 四个标签组织
3. 每个标签先覆盖：
   - 列表
   - 详情
   - 基础操作
   - 简单筛选
4. 页面顶部接入轻量统计摘要
5. 将入口纳入当前导航的 `配置中心`

---

## 非目标

1. 本轮不做完整 CRUD 覆盖
2. 本轮不做复杂统计图表
3. 本轮不做批量操作
4. 本轮不接真实渠道发送联调
5. 本轮不做推送事件编排或高级运营能力

---

## 信息架构

### 导航归类

当前侧边栏已经按四组分区：

- 工作台
- 情报数据
- 知识资产
- 配置中心

本轮新增：

- 在 `配置中心` 下增加 `推送管理`

设计意图：

- 推送管理更接近“系统输出与配置能力”，而不是情报链路主干数据
- 它和 `系统设置` 放在一起更符合用户心智

---

## 页面结构

新页面路由建议：

- `/push`

页面整体结构：

1. 页面标题区
2. 顶部统计卡
3. 标签页主工作区

### 1. 页面标题区

包含：

- 页面标题：`推送管理`
- 副标题：说明这是对推送渠道、任务、记录、模板的统一管理台

不放复杂工具栏，不在这一层堆太多按钮。

### 2. 顶部统计卡

统计来自 `/api/v1/push/stats`

本轮只展示最有价值的摘要：

- 总任务数
- 启用任务数
- 总推送记录数
- 成功率

设计原则：

- 保持轻量
- 不做图表
- 只作为进入管理台时的状态感知

### 3. 标签页主工作区

使用 `Tabs` 组织 4 个主标签：

- `渠道`
- `任务`
- `记录`
- `模板`

切换逻辑：

- 默认进入 `任务`
- 因为任务最接近日常管理动作

## 交互设计

### 列表形态

这次不建议四个标签全都做成完全不同的视觉风格。

建议：

- `任务`、`记录` 使用表格或强结构列表
- `渠道`、`模板` 使用卡片列表或轻表格

理由：

- 任务和记录更偏运行态信息，表格更利于扫读
- 渠道和模板更偏配置对象，卡片更容易读“对象感”

### 详情查看

统一使用右侧 `Sheet` 抽屉，而不是每个标签都跳转详情页。

理由：

- 更符合当前仓库已有模式
- 开发成本更低
- 管理台操作更流畅

### 基础操作

操作应尽量贴近列表项，不要把所有动作塞进详情里：

- 渠道：启用 / 停用
- 任务：启用 / 停用 / 手动触发
- 记录：失败重试
- 模板：预览

需要二次确认的动作：

- `重试`
- `手动触发`

---

## 视觉方向

延续当前系统已形成的深色工作台方向，不单独发明新视觉语言。

视觉重点：

1. 顶部统计卡与标签页之间层级清晰
2. `任务` 标签要最易扫读
3. 状态 badge 清晰可辨
4. hover 不制造布局抖动
5. 桌面和移动端都能稳定使用

结合现有 UIPro 方向，本页应保持：

- 深色背景
- 高对比文本
- 绿色作为正向成功色
- 弱 glow，仅用于重点，不要满屏发光

## 数据流设计

页面级数据流：

1. 页面载入后请求统计摘要
2. 标签页各自请求自己的列表数据
3. 用户触发基础操作后，回刷当前标签和必要统计

刷新策略：

- `任务` 启停、触发后：
  - 刷新任务列表
  - 刷新统计摘要
  - 必要时刷新记录列表
- `记录` 重试后：
  - 刷新记录列表
  - 刷新统计摘要
- `渠道` 启停后：
  - 刷新渠道列表

## 响应式策略

### Desktop

- 顶部统计卡 4 列
- 标签内容区为主工作区
- 详情用右侧抽屉

### Mobile

- 顶部统计卡堆叠或 2 列
- Tabs 保持可点、可横向容纳
- 列表内容优先保持关键信息
- 抽屉继续使用

不做移动端专属复杂交互。

## 风险与控制

### 风险 1：一次接太多 API，页面复杂度膨胀

控制方式：

1. 只做最小可用范围
2. 不在本轮补全 CRUD
3. 不在本轮做高级统计

### 风险 2：四个标签实现方式过于分裂

控制方式：

1. 页面层统一组织
2. 详情交互统一用 `Sheet`
3. 尽量复用现有 `FilterBar`、`ConfirmDialog`、`Tabs`

### 风险 3：导航挂载导致信息架构混乱

控制方式：

1. 明确将 `推送管理` 放在 `配置中心`
2. 不在本轮再改其它导航分组

---

## 验收标准

本轮完成后，应满足：

1. 新增 `/push` 页面
2. 页面已纳入侧边栏 `配置中心`
3. 顶部统计卡接真实 `/push/stats`
4. `渠道 / 任务 / 记录 / 模板` 四个标签均可访问
5. 每个标签至少具备：
   - 列表
   - 详情
   - 基础操作
   - 简单筛选
6. `apps/web` 的 `lint` / `build` 持续通过

---

## 本轮建议实现优先级

1. 先把类型、API、hooks 铺好
2. 先做统计和 `任务` 标签
3. 再补 `渠道 / 记录 / 模板`
4. 最后挂入导航并统一验收
